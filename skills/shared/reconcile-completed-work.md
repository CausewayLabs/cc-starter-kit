# Shared routine — Reconcile completed work

A reusable routine that detects which plan steps / streams were finished since the
active plan was generated, and marks them complete. Called from two places:

- **`/next-plan`** (precursor — runs *before* selecting the next plan; authoritative,
  guaranteed to run because you can't pick new work without it).
- **`/wrapup`** (warm — runs at end of session while the agent still has live context;
  higher fidelity but skippable).

Both call this same routine. It is **idempotent** — re-running it marks nothing new.

---

## What it touches (two lanes — keep them separate)

**In scope (mechanical, this routine):**
- `memory/current-plan.md` — step checkboxes `- [ ]` → `- [x]`.
- `memory/plans/*.md` (master/multi-stream plans, if present) — the **Status Snapshot**
  table + per-stream `## Stream X — … — <status>` header markers.
- The matching feature-log entry, **only when** a plan step carries a
  `[source: architecture/<subsystem>-log.md / <Category>]` reference — flip that one
  entry's checkbox to `[x]` (the verbatim step text matches the log line).

**Out of scope (judgment, not here):**
- *Promoting* a finding/decision into the architecture doc or Codebase Guide — that's the
  promotion test, owned by `/memory-log-review`. This routine flips checkboxes; it does not
  graduate knowledge.
- Logging brand-new work items. (That's the organic-capture heuristic.)

---

## Steps

### R1 — Find the baseline

Read `memory/current-plan.md` and look for a base-SHA stamp in the header:

```
<!-- base-sha: <40-char-sha> -->
```

- **If present:** the diff window is `git log <base-sha>..HEAD --oneline`.
- **If absent** (older plan, or stub): fall back to the `Generated by /next-plan on
  <YYYY-MM-DD>` date and use `git log --since=<YYYY-MM-DD> --oneline`. Note to the user that
  the stamp was missing so the window is approximate.
- If neither exists, there is nothing to reconcile — say so and return.

### R2 — Gather the evidence

Run (read-only):

```
git log <base-sha>..HEAD --oneline        # commits during this plan
git diff <base-sha>..HEAD --stat          # files touched (corroboration)
```

Keep the raw output in the sandbox if large — only the mapping conclusion needs to enter context.

### R3 — Map commits → plan items

For each unchecked step in `current-plan.md` and each open stream/task in the master plan,
decide whether the commits provide **clear evidence** it's done. Strong signals:
- A commit whose scope/message names the subsystem + the thing the step describes
  (`feat(stream-d): …`, `fix(auth): token refresh …`).
- `--stat` shows the files the step would have to touch were changed.
- The step text already appears as `[x]` (already marked — skip, idempotent).

Be conservative. Cold inference is lossy — a vague commit is **not** evidence.

### R4 — Propose, then confirm (do NOT silently flip)

Present a short list:

> Looks completed since this plan started:
> - Step 2 (AppShell logoUrl) — `a1b2c3d feat(portal): add logoUrl to AppShell`
> - Stream D — `e2b3cae … multi-tenant primitives`
>
> Mark these complete? (Anything I should leave open?)

Wait for confirmation. The user may reject items (cold inference was wrong) or add ones the
commits didn't make obvious. Honor their edits.

**Exception:** when called from `/wrapup` and the agent did the work *this session* (warm
context, high confidence), it may state what it's marking and proceed without a separate
confirmation round — but still show the list.

### R5 — Mark the confirmed items

- `current-plan.md`: flip `- [ ]` → `- [x]` on confirmed steps. Append `(completed:
  YYYY-MM-DD)` if the plan's step format carries dates.
- `memory/plans/*.md`: update the Status Snapshot row(s) and the `## Stream X` header
  marker(s) to the new state (✅ Complete / 🔜 Next / ⏸️ Deferred), with the date.
- For steps with a `[source:]` ref: flip that one feature-log entry to `[x]` and add
  `(completed: YYYY-MM-DD)` per the project's Feature Log Format.

### R6 — Hand back

Return a one-line summary of what was marked (or "nothing to reconcile"). The caller
continues its own flow.
