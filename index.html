1The Golden Rule: Use Git Liberally

If you remember one thing from this whole document, make it this one. And here's the good news — you don't actually have to learn git. You just have to know what it's for, and tell Claude to do it.

Commit early, commit often. Claude Code edits real files on your machine. It is fast, capable, and occasionally confidently wrong. Git is your undo button, your safety net, and your time machine. The whole trick is to just ask Claude — "commit this," "save my progress" — and it handles the actual commands. You never need to memorize them.

First, git vs. GitHub — they're not the same thing

People use these interchangeably, but they're different:

  • Git is the tool running on your computer that tracks the history of your files — every saved snapshot lives right there on your machine, no internet needed.
  • GitHub is a website where you can push a copy of that history to the cloud — for backup, or to share with someone (it's how you got this kit).

For day-to-day work, git is the part that matters. GitHub is just where it goes to be backed up or shared later.

The only concept you truly need: the commit

A commit is a saved snapshot of your whole project at a moment in time. That's it. Every time you commit, git remembers exactly what everything looked like — and you can always go back.

This is your undo button. Say you commit, then ask Claude to add a feature and it makes a mess — because you have that snapshot, you can rewind to exactly how things were a moment ago, in one step, nothing lost. Without a recent commit, there's nothing to rewind to.

So the habit is simple: when you've made progress you'd hate to lose, tell Claude "commit this." Do it often. Small, frequent snapshots mean any mistake is never more than a moment's worth of work away from being erased.

If something goes sideways, you can always undo — just ask. Say "undo that, go back to the last commit" or "revert the last change." As long as you've been committing along the way, Claude can walk it back for you. You don't need to know the command — you just need to know that the safety net exists, and that committing often is what keeps it under you.

A few more words worth recognizing (so you know what to ask for)

You don't need to run these — but knowing the words exist means you can ask Claude for them by name when the moment comes:

WordWhat it's for
diff"Show me what changed." Ask for a diff when you want to see exactly what Claude edited before committing.
branchA parallel copy to try something risky without touching your main work. "Make a branch before you try this."
worktreeLike a branch, but checked out in its own folder — handy for working on two things at once. "Spin this up in a worktree."
resetThe big undo — throw away changes and snap back to a previous commit. "Reset to the last commit."

That's the whole vocabulary. Commit is the one that matters; the rest are just things to ask for by name when you need them.

Make it automatic: the /wrapup skill. When you're done for the day, just run /wrapup. It updates your project docs and commits everything in one move — so you end every session with a clean snapshot saved, without even having to remember to ask. It's the easiest way to build the "commit often" habit: let the skill do it for you at the end of each session.

2Skills 101 — What's In This Kit

A "skill" is a folder of instructions Claude loads on demand. You trigger one by typing a slash command (like /brainstorm) or just describing what you want.

Each skill is just a SKILL.md file living in ~/.claude/skills/<name>/. When your words match a skill's description, Claude pulls in that playbook and follows it. You don't install anything special — you drop the folders in place (see Install) and they're available everywhere.

This kit hands you a few workflows that build on each other:

/new-project — scaffold a clean repo + memory system the memory system — how the project stays legible /brainstorm → the agent pipeline — build it for you /handoff + /handoff-resume — never lose your place

3/new-project — Start Greenfield The Right Way

The front door for any brand-new app or repo. It builds the skeleton so you don't have to think about structure.

/new-project "set up a new project" · "spin up a new app"

It creates a clean project root with a deliberate idea baked in: separate the project's "mind" from its "body."

  • The root stays code-free — it holds the memory system, a CLAUDE.md, a .context.md router, git, and a run.bat one-command launcher.
  • All real code lives one folder down (default app/) so build clutter never tangles with your docs.
  • It runs git init and writes a sensible .gitignore for you.
  • Under the hood it calls /memory-init to install the memory system — that's section 4, and it's the part worth understanding well.

It deliberately doesn't scaffold the framework itself (no create-next-app) — that's your next step, on purpose. You get an oriented shell, not an empty folder.

my-project/
├── .context.md      # the "read me first" router
├── CLAUDE.md        # the agent operating manual
├── run.bat          # single-command launcher
├── memory/          # the whole memory system (see §4)
└── app/             # ← all your actual code lives here

4The Memory System — What Goes Where

This is the part that pays off most over time. The memory system is how a project stays understandable — to you, and to a fresh Claude session that has zero memory of last week.

First, the relief: you don't have to manage this by hand. When you run /new-project (or /memory-init), every one of these files is created pre-seeded with the instructions Claude needs to maintain them itself. The CLAUDE.md it writes literally tells Claude how to log tasks, where to file decisions, and when to promote things. In normal use you just work, and Claude keeps the memory system tidy in the background. So why read the rest of this section? Because knowing the structure lets you spot when Claude drifts — files a decision in the wrong place, or lets a log entry that should be permanent truth rot as a receipt — and course-correct it with one sentence. You're the editor, not the filing clerk.
The one mental model: separate the permanent truth of the system (what it is, why it's shaped this way, what must always hold) from the disposable history of work (what's queued, what just changed). The first is the charter — it's load-bearing and you protect it. The second is receipts — useful logs, but throwaway. The skill below is keeping these from blurring together. 
my-project/
├── .context.md                     # ROUTER — read first, indexes everything
├── CLAUDE.md                       # OPERATING MANUAL — how we work here
└── memory/
    ├── architecture/
    │   ├── architecture.md          # THE SPINE / CHARTER — what & why
    │   ├── <subsystem>.md           # subsystem design (charter, extended)
    │   └── <subsystem>-log.md       # feature log — RECEIPTS (queued/done)
    ├── Codebase Guide.md            # where code lives + invariants
    ├── Glossary.md                  # domain vocabulary
    ├── current-plan.md              # the active plan (/next-plan writes it)
    ├── task-feedback.md             # tuning log for auto-capture
    ├── handoffs/                    # session-continuity docs (see §6)
    └── external/                    # distilled outside sources

The five files you'll actually think about

.context.md — the router (project root)

The "read me first" for the whole project. It's pointers and orientation, not deep content: a one-line tagline, index tables that say where every other file lives, and a Commands table (install / run / test / build). Any fresh session reads this first and instantly knows the lay of the land.

Belongs here: the tagline, the file index, the run/test/build commands. Not here: actual design reasoning or task lists — those just get linked from here.

CLAUDE.md — the operating manual (project root)

Claude reads this at the start of every session. It's the "how we work in this repo" rulebook: stable conventions like the planning workflow, how tasks get captured, the feature-log format, the session-start checklist. It is deliberately stable — it describes process, not state.

✓ Belongs in CLAUDE.md

  • Conventions & rules ("always read .context.md first")
  • How planning works, how tasks are logged
  • Formats the project follows
  • Coding standards specific to this repo

✗ Does NOT belong

  • What you're building right now (that's a plan/log)
  • A changelog or roadmap
  • The actual architecture (that's the spine)
  • Anything that changes week to week

Rule of thumb: if it would be true six months from now regardless of what you build, it's CLAUDE.md material.

memory/architecture/architecture.md — the spine (charter)

The conceptual constitution of the project. Two things at the top level: a 2–3 sentence System Purpose (what this is and why it exists) and a list of subsystems. It answers what the system is and why it's shaped the way it is — not how the code is laid out.

It's called the charter because every decision must be congruent with it. If a request conflicts with the charter, the right move is to stop and surface the tension — not quietly drift. This is the document an agent trusts above all others.

<subsystem>.md + <subsystem>-log.md — subdocs & their logs

As real subsystems emerge (auth, billing, the rendering engine…), each gets a pair of files in memory/architecture/. You don't pre-create these — they're born when a subsystem becomes real.

FileWhat it holds
<subsystem>.md
(charter)		 Prose describing the subsystem's design and the why behind it: its purpose, the boundary it owns, the major decisions and their rationale. This is permanent truth — read it before touching that area.
<subsystem>-log.md
(receipts)		 A checklist of work items — planned and completed tasks, dated, with completed ones carrying a short "decision: chose X over Y because Z" note. This is the "what's queued / what just changed" log. Receipts, not documentation.

A log entry looks like this:

## Core Rendering
- [x] (planned: 2026-01-10, completed: 2026-01-18) Implement draw loop
      — decision: chose retained-mode over immediate-mode because deep
      widget trees caused jitter redrawing every frame.
- [ ] (planned: 2026-02-05) Add animation easing presets

memory/Codebase Guide.md — where + what-must-hold

The companion to the spine. Where architecture answers why, this answers where the code lives (the layout) and what invariants must always hold (rules the system depends on — "all timestamps are UTC", "the API layer never imports from the UI layer").

The promotion test — the habit that keeps docs healthy. Feature logs are receipts; the charter is truth. When a completed task contains a real decision, ask: "if this sentence vanished from the architecture doc, could a future agent steer the system wrong?" If yes → promote it up into the subsystem doc (rationale) or the Codebase Guide (invariants). If no → leave it in the log as a receipt. The goal: an agent reading only the charter + Codebase Guide, with no logs at all, should still make good decisions. (The /memory-log-review skill walks the logs and applies this test in batch.)

/next-plan — deciding what to build next "what's next" · "plan my next session"

Once your project has a memory system with some work logged in it, /next-plan is how you start each session. It reads the spine, the subsystem docs, and the feature logs, then picks the next sensibly-sized chunk of work and writes it to current-plan.md as a step-by-step plan you can begin executing right away. It even tells you why it chose that work over everything else.

Think of it as the everyday counterpart to the big /brainstorm pipeline: use the pipeline to build a whole new feature from scratch, and /next-plan to keep chipping away at what's already logged — without having to remember where you left off or decide what matters most.

5/brainstorm → The Agent Pipeline

This is the headline feature. You start with a fuzzy idea; a relay team of AI agents turns it into working code — each agent doing one job well, with you in the driver's seat at the start.

Step one: /brainstorm "let's brainstorm" · "I have an idea to build"

Claude becomes a thought partner, not a coder. It asks sharp clarifying questions, reflects your idea back, plays devil's advocate, and offers options with trade-offs — all at the idea level, never jumping to implementation.

When you're happy, it writes a clean feature plan to plans/your-feature.md and asks permission to hand off. Nothing proceeds without your explicit "go."

Once you approve, it becomes the coordinator and runs four phases automatically — each a specialist agent. It only stops to talk to you if something genuinely needs a decision.

Phase 1

Architect

Turns the feature plan into a concrete development plan: tasks, structure, approach.

→
Phase 2

Opus Review

A heavyweight model reviews & auto-corrects the plan, surfacing only real blockers.

→
Phase 3

Team Lead

Spawns developer agents for each task and manages execution order.

→
Phase 4

QA Review

Validates everything built against the original spec & success criteria.

The mental model: you are the product owner. You spend your energy on the brainstorm — getting the idea right — and the pipeline handles plan → review → build → QA. Garbage in, garbage out: a good brainstorm is what makes the rest sing.

Bonus: /orchestrator "you coordinate, agents code"

A lighter-weight mode for when you don't need the full pipeline. Claude stops writing code itself and instead decomposes the work, dispatches developer agents, and reviews their output as the QA layer. Great for medium tasks where you want delegation without the full brainstorm ceremony.

6/handoff + /handoff-resume — Never Lose Your Place

Long sessions fill up Claude's context window — the amount it can "hold in its head" at once. As it fills, responses get slower and details slip. These two skills let you reset to an empty window with zero loss of momentum.

When to reach for it — the 50% rule. Ryan's habit: fire /handoff at around 50% context-window usage. Don't wait until it's nearly full and degraded — hand off while the session is still sharp and remembers everything clearly. Half-full is the sweet spot.
0%
100%

↑ The white line is ~50% — that's your cue to /handoff, not the far right edge.

/handoff "context is filling up" · "save my place"

Writes a dense, self-contained handoff document capturing everything a fresh agent needs: what the project is, what you were doing and why, exactly where things stand, what's left (as a checklist), key files, and any gotchas or decisions. Saved to memory/handoffs/YYYY-MM-DD-HHMM.md.

/handoff-resume "pick up where I left off"

The first thing you run in a fresh session. It finds the most recent handoff doc, reads it plus the project's orientation files, gives you a 4-line summary of where you are — then immediately gets back to work on the first unchecked item.

The loop: /handoff  →  /clear  →  /handoff-resume. Your context window resets to empty, but your work picks up exactly where it left off. /clear is the built-in command that wipes the session's memory — the handoff doc is what survives it.

7A Day In The Life — Putting It Together

Here's how the pieces chain into one smooth workflow, from empty folder to shipped feature.

  1. Scaffold it. Run /new-project — clean repo, git initialized, memory system installed.
  2. Commit the skeleton. "Commit this as the initial scaffold." Now you have a safe baseline.
  3. Fill in the charter. Write a couple sentences of System Purpose in architecture.md and your run command in .context.md. Five minutes, huge payoff later.
  4. Brainstorm the feature. Run /brainstorm. Answer the questions honestly — this is where your real input matters most.
  5. Approve & let it run. Say "go." The pipeline architects → reviews → builds → QAs while you watch.
  6. Commit the result. Like what it built? git commit. Don't? git reset --hard and refine the brainstorm.
  7. Halfway through context? /handoff → /clear → /handoff-resume and keep rolling, fresh and fast.
Day two and beyond: you won't re-scaffold every time. Most sessions just start with /next-plan ("what should I work on next?") or /handoff-resume — then you work, commit, and finish with /wrapup. The full brainstorm pipeline only comes out when you're building a brand-new feature.

8Install — Dropping These Skills In

Two ways to do this: the easy way (let Claude do it) or the manual way (copy the folders yourself). Either is fine.

The easy way — let Claude install them for you

Open Claude Code anywhere and paste it this:

Read https://github.com/CausewayLabs/cc-starter-kit and help me
install these skills into my Claude skills folder.

Because the repo is public, Claude can read it directly and then walk you through copying the skill folders into place. If anything goes sideways, just keep talking to it — "it didn't work," "I'm on a Mac," "where do these go?" — and it'll adapt.

Where skills go (if you'd rather do it by hand)

OSSkills folder
WindowsC:\Users\<you>\.claude\skills\
macOS / Linux~/.claude/skills/

Each skill is its own subfolder containing a SKILL.md. Drop the folders in, restart Claude Code, done.

The manual way — clone and copy

# 1. Clone the repo I shared with you
git clone <the-repo-url> cc-starter-kit
cd cc-starter-kit

# 2. Copy the skill folders into your Claude skills directory
#    (Windows PowerShell)
Copy-Item -Recurse skills\* $env:USERPROFILE\.claude\skills\

#    (macOS / Linux)
cp -r skills/* ~/.claude/skills/

# 3. Restart Claude Code, then try it:
/new-project
Staying up to date: this is a public GitHub repo, so any time I improve a skill you can grab the latest — just tell Claude "git pull the starter-kit repo" or re-run the easy-way step above. The skill files use portable paths (~/.claude/..., ~/Projects/...) so they should work on your machine out of the box — but if anything assumes a setup you don't have, just ask Claude to "adapt these skills to my machine."
Lost? Ask Claude itself. Type /help, or just say "what skills do I have?" or "how do I use the brainstorm skill?" The best documentation is the assistant sitting right in front of you.
