1The Golden Rule: Use Git Liberally

If you remember one thing from this whole document, make it this one.

Commit early, commit often. Claude Code edits real files on your machine. It is fast, capable, and occasionally confidently wrong. Git is your undo button, your safety net, and your time machine. A clean commit before you start means any mess is one command away from being erased.

Why it matters more with an AI agent

  • It moves fast. Claude can touch a dozen files in one turn. Without commits, "undo" doesn't exist.
  • It lets you be bold. When you can always roll back, you can let Claude try the ambitious thing.
  • It gives Claude context. A clean history and good messages help future sessions understand what happened and why.

The rhythm to build into muscle memory

# Before you start a task — make sure you're clean
git status

# Made progress you'd hate to lose? Snapshot it.
git add -A
git commit -m "Add login form and validation"

# Claude went sideways? Throw away uncommitted changes:
git restore .            # discard unstaged edits
git reset --hard HEAD    # nuke everything back to last commit

# Trying something risky? Branch first, merge only if it works.
git checkout -b try-new-idea

You don't need to be a git wizard. status, add, commit, and reset --hard cover 95% of daily life.

Tip: Just tell Claude to do it — "commit this with a good message" or "create a branch before you start." It knows git. Your job is to remember to ask.

2Skills 101 — What's In This Kit

A "skill" is a folder of instructions Claude loads on demand. You trigger one by typing a slash command (like /brainstorm) or just describing what you want.

Each skill is just a SKILL.md file living in ~/.claude/skills/<name>/. When your words match a skill's description, Claude pulls in that playbook and follows it. You don't install anything special — you drop the folders in place (see Install) and they're available everywhere.

This kit hands you four workflows that work together:

/new-project — scaffold a clean repo /brainstorm — refine an idea into a plan the agent pipeline — build it for you /handoff + /handoff-resume — never lose your place

3/new-project — Start Greenfield The Right Way

The front door for any brand-new app or repo. It builds the skeleton so you don't have to think about structure.

/new-project "set up a new project" · "spin up a new app"

It creates a clean project root with a deliberate idea baked in: separate the project's "mind" from its "body."

  • The root stays code-free — it holds notes, docs, a CLAUDE.md (project instructions), git, and a run.bat one-command launcher.
  • All real code lives one folder down (default app/) so build clutter never tangles with your docs.
  • It runs git init and writes a sensible .gitignore for you.
  • It deliberately doesn't scaffold the framework — that's your next step, on purpose.

You end up with an oriented, ready-to-build shell instead of an empty folder and a blank stare.

What you get

my-project/
├── .context.md      # the "read me first" for any agent
├── CLAUDE.md        # project-specific instructions
├── run.bat          # single-command launcher
├── memory/          # notes, logs, plans, handoffs
└── app/             # ← all your actual code lives here

4/brainstorm → The Agent Pipeline

This is the headline feature. You start with a fuzzy idea; a relay team of AI agents turns it into working code — each agent doing one job well, with you in the driver's seat at the start.

Step one: /brainstorm "let's brainstorm" · "I have an idea to build"

Claude becomes a thought partner, not a coder. It asks sharp clarifying questions, reflects your idea back, plays devil's advocate, and offers options with trade-offs — all at the idea level, never jumping to implementation.

When you're happy, it writes a clean feature plan to plans/your-feature.md and asks permission to hand off. Nothing proceeds without your explicit "go."

Once you approve, it becomes the coordinator and runs four phases automatically — each a specialist agent. It only stops to talk to you if something genuinely needs a decision.

Phase 1

Architect

Turns the feature plan into a concrete development plan: tasks, structure, approach.

→
Phase 2

Opus Review

A heavyweight model reviews & auto-corrects the plan, surfacing only real blockers.

→
Phase 3

Team Lead

Spawns developer agents for each task and manages execution order.

→
Phase 4

QA Review

Validates everything built against the original spec & success criteria.

The mental model: you are the product owner. You spend your energy on the brainstorm — getting the idea right — and the pipeline handles plan → review → build → QA. Garbage in, garbage out: a good brainstorm is what makes the rest sing.

Bonus: /orchestrator "you coordinate, agents code"

A lighter-weight mode for when you don't need the full pipeline. Claude stops writing code itself and instead decomposes the work, dispatches developer agents, and reviews their output as the QA layer. Great for medium tasks where you want delegation without the full brainstorm ceremony.

5/handoff + /handoff-resume — Never Lose Your Place

Long sessions fill up Claude's context window. These two skills let you reset with zero loss of momentum.

/handoff "context is filling up" · "save my place"

Writes a dense, self-contained handoff document capturing everything a fresh agent needs: what the project is, what you were doing and why, exactly where things stand, what's left (as a checklist), key files, and any gotchas or decisions. Saved to memory/handoff/YYYY-MM-DD-HHMM.md.

/handoff-resume "pick up where I left off"

The first thing you run in a fresh session. It finds the most recent handoff doc, reads it plus the project's orientation files, gives you a 4-line summary of where you are — then immediately gets back to work on the first unchecked item.

The loop: /handoff  →  /clear  →  /handoff-resume. Your context window resets to empty, but your work picks up exactly where it left off. Use this whenever Claude starts feeling sluggish or forgetful.

6A Day In The Life — Putting It Together

Here's how the pieces chain into one smooth workflow, from empty folder to shipped feature.

  1. Scaffold it. Run /new-project to get a clean, git-initialized repo with the right bones.
  2. Commit the skeleton. "Commit this as the initial scaffold." Now you have a safe baseline.
  3. Brainstorm the feature. Run /brainstorm. Answer the questions honestly — this is where your real input matters most.
  4. Approve & let it run. Say "go." The pipeline architects → reviews → builds → QAs while you watch.
  5. Commit the result. Like what it built? git commit. Don't? git reset --hard and refine the brainstorm.
  6. Running low on context? /handoff → /clear → /handoff-resume and keep rolling.

7Install — Dropping These Skills In

Skills live in one folder. Getting them is a clone-and-copy.

Where skills go

OSSkills folder
WindowsC:\Users\<you>\.claude\skills\
macOS / Linux~/.claude/skills/

Each skill is its own subfolder containing a SKILL.md. Drop the folders in, restart Claude Code, done.

The clone-and-copy

# 1. Clone the repo I shared with you
git clone <the-repo-url> cc-starter-kit
cd cc-starter-kit

# 2. Copy the skill folders into your Claude skills directory
#    (Windows PowerShell)
Copy-Item -Recurse skills\* $env:USERPROFILE\.claude\skills\

#    (macOS / Linux)
cp -r skills/* ~/.claude/skills/

# 3. Restart Claude Code, then try it:
/new-project
How I'm sending this to you: a private GitHub repo. Clone it once, and any time I improve a skill you just run git pull to get the update. The skills reference some of my personal paths (like D:\Projects) — feel free to edit the SKILL.md files to match your own setup. When in doubt, just ask Claude: "adapt these skills to my machine."
Lost? Ask Claude itself. Type /help, or just say "what skills do I have?" or "how do I use the brainstorm skill?" The best documentation is the assistant sitting right in front of you.
